# Auto Buyback & Burn

Turn every credit purchase into buy pressure for your token — automatically, on-chain, with zero manual steps.

When you enable Auto Buyback & Burn, a percentage of every top-up your users make is routed to a dedicated wallet that automatically swaps accumulated fees to your token and burns them. No cron jobs to manage. No multisig to coordinate. No manual execution.

<Callout type="info">
  **This is a zero-fee feature.** EarnKit takes 0% of your revenue. The buyback allocation comes from the developer's share — you decide how much of your own revenue to route into buyback.
</Callout>

---

## How It Works

```
User tops up → Split contract distributes fees
                ├── (100 - buybackPercent)%  → your developer wallet (immediate)
                └── buybackPercent%          → buyback wallet
                                                  └── threshold crossed?
                                                        ├── yes → swap to your token → burn
                                                        └── no  → accumulates
```

### Step by step

1. **Agent creation** — you set a buyback percentage (e.g. 10%) and your token's contract address. EarnKit deploys a SplitV2 contract on Base that permanently routes that share to a Turnkey-managed buyback wallet.

2. **Fees accumulate** — every time a user pays (ETH, USDC, or MINIDEV), your portion goes to your wallet instantly and the buyback share lands in the Turnkey wallet.

3. **Threshold check** — after each distribution, EarnKit checks the wallet's swappable USD value (gas reserve excluded). Once it crosses your configured threshold, a buyback execution is queued.

4. **Swap & burn** — assets are withdrawn from the Splits Warehouse, swapped to your token via Uniswap V3 on Base, then burned by calling `burn()` on your token contract (or transferred to `0x000...dEaD` if your token doesn't implement `burn()`).

---

## Enabling It

Buyback is configured at agent creation time in the EarnKit dashboard.

| Setting | Description |
|---------|-------------|
| **Buyback %** | Percentage of fees routed to the buyback wallet (e.g. `10` = 10%) |
| **Token address** | The ERC-20 token to buy and burn on Base |
| **USD threshold** | Minimum swappable balance before execution triggers (e.g. `$100`) |
| **Recovery address** | Where to sweep funds if execution fails after all retries |

<Callout type="warning">
  The buyback wallet address is embedded in the SplitV2 contract **at deploy time** and cannot be changed. Choose your settings carefully before creating the agent.
</Callout>

## Revenue Split Example

With a 10% buyback allocation on a $100/month agent:

```
$100 in user top-ups
  ├── $90  → your developer wallet
  └── $10  → buyback wallet → swapped to your token → burned
```

Platform fee: **0%**. You keep everything not allocated to buyback.

## Execution States

| State | What's happening |
|-------|-----------------|
| `QUEUED` | Threshold crossed — execution is scheduled |
| `WITHDRAWING` | Pulling assets from Splits Warehouse to the buyback wallet |
| `SWAPPING` | Swapping ETH / USDC / MINIDEV → your token via Uniswap V3 |
| `BURNING` | Calling `burn()` or transferring to `0x000...dEaD` |
| `COMPLETED` | Tokens burned successfully |
| `RETRYING` | A step failed — retrying with backoff (up to 5 attempts: 1m, 5m, 30m, 2h, 12h) |
| `FAILED` | All retries exhausted — use "Recover stranded funds" to retrieve assets |

---

## Supported Input Tokens

The buyback wallet accepts ETH, USDC, and MINIDEV as inputs. Each is independently swapped to your buyback token via the best available Uniswap V3 route (direct or two-hop via WETH).

If one asset's swap fails, the others still proceed — a failed USDC route will not block ETH or MINIDEV from being swapped and burned.

**Special case:** if fees arrive already denominated in your buyback token (e.g. a MINIDEV agent with MINIDEV buyback configured), no swap is needed. The executor detects the token balance after the warehouse withdraw and proceeds directly to burn.

## Dashboard Controls

| Action | Description |
|--------|-------------|
| **Trigger now** | Manually queue a buyback if the threshold is already crossed |
| **Disable** | Pause auto-buyback — funds stay in the wallet and can be recovered |
| **Recover stranded funds** | Sweep all assets from the buyback wallet to your recovery address |

## Security Design

- **Immutable split** — the buyback wallet address is baked into the SplitV2 contract at deploy time. There is no admin key that can redirect funds post-deployment.
- **Operator gas drip** — the buyback wallet needs ETH for gas. A platform-operated Turnkey wallet drips a fixed 0.0001 ETH reserve before each execution. The Turnkey policy restricts transfers to known buyback wallet addresses only.
- **Gas reserve excluded** — the 0.0001 ETH gas reserve is subtracted before computing the swappable balance, so the threshold check is never falsely triggered by the gas drip alone.
- **Per-asset isolation** — each input token is swapped in its own try/catch, preventing one failed route from blocking others.

## FAQ

**Can I change the buyback percentage after creating the agent?**
No. The split is on-chain and immutable. You would need to create a new agent with updated settings.

**What happens if my token doesn't have a `burn()` function?**
EarnKit transfers the tokens to `0x000000000000000000000000000000000000dEaD` (the dead address).

**What if the buyback wallet runs out of ETH for gas?**
The platform drips 0.0001 ETH before each execution attempt. If the drip fails, the execution is retried with backoff.

**Can I use any ERC-20 as my buyback token?**
Yes — any ERC-20 on Base with a Uniswap V3 pool. The executor finds the best route automatically.

**What if execution fails permanently?**
After 5 retries, the execution status is set to `FAILED`. Use **Recover stranded funds** in the dashboard to sweep all assets to your recovery address.
